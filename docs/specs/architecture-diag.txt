# ğŸ§± **Layerâ€‘byâ€‘Layer Explanation**

## **1. Web App UI**
Your user interacts with a simple interface:
- Project name  
- Path  
- Language stack  
- Include Jira?  
- Include CI?  
- Include repo?  

The UI sends a structured request to the backend.

---

## **2. Web App Backend**
This layer:
- Normalizes the request  
- Builds an **OrchestrationSpec**  
- Sends it to the Orchestration Engine  

This is where you enforce:
- Authentication  
- Org defaults  
- User permissions  

---

## **3. Orchestration Engine**
This is the **brain** of the system.

It:
- Validates the spec  
- Selects the correct agents  
- Builds a deterministic execution plan  
- Executes agents in order  
- Collects outputs  
- Sends results back to the web app  

This is where your multiâ€‘agent orchestration expertise shines.

---

## **4. Agent Layer**
Each agent is a **singleâ€‘responsibility module**.

Agents include:
- ProjectSpecCollectorAgent  
- SpecValidatorAgent  
- TemplateSelectorAgent  
- PlanBuilderAgent  
- FilesystemScaffolderAgent  
- GitRepoProvisionerAgent  
- WorkflowGeneratorAgent  
- JiraSpaceProvisionerAgent  
- SummaryReporterAgent  

Each agent receives:
- `plan_step`
- `project_spec`
- `context`

Each agent returns:
- `status`
- `output`
- `errors`

---

## **5. Integration Layer**
This layer handles external systems:
- Jira REST API  
- GitHub/GitLab API  
- CI/CD providers  
- Cloud providers (optional)  

Agents never call APIs directly â€” they call the Integration Layer.

---

## **6. Local System / Filesystem**
Used by:
- FilesystemScaffolderAgent  
- GitRepoProvisionerAgent  

This is where:
- Directories are created  
- Files are scaffolded  
- Git repos are initialized
